# Spring Data AOT and native-image

Open this reference when the ordinary repository-abstraction path in [SKILL.md](../SKILL.md) is not enough and the task depends on AOT or native-image repository details.

Use this branch only when Spring AOT is actually enabled for the application, whether the target is a native image or a JVM build that still uses AOT processing.

## What AOT changes

Spring Data AOT precomputes repository metadata and type access paths so native-image or AOT builds do not rely on broad runtime reflection.

Use this branch only when the application actually builds with AOT enabled.

## Repository eligibility rule

AOT support works best with imperative repository methods that stay within supported query and projection patterns.

Reactive repositories are outside this repository-AOT path.

For JPA specifically, AOT repository generation excludes methods that accept `ScrollPosition` and dynamic projection methods.

Good fit:

- derived queries
- declared `@Query` methods
- projections with stable return shapes
- pageable and sortable repository methods

Riskier fit:

- dynamic reflection-heavy patterns
- open-ended runtime proxy behavior
- store-specific features not covered by the target module's AOT support

## Configuration shape

These properties are optional tuning flags once the application already builds with AOT enabled.

```properties
spring.aot.data.accessors.enabled=true
spring.aot.data.accessors.include=com.example.customer.Customer
spring.aot.data.accessors.exclude=com.example.legacy.LegacyCustomer
spring.aot.repositories.enabled=true
spring.aot.repositories.metadata.enabled=true
spring.aot.jpa.repositories.enabled=true
spring.aot.jpa.repositories.use-entitymanager=true
spring.aot.mongodb.repositories.enabled=true
```

These examples show tuning flags only. Repository generation and metadata contribution are already enabled by default once AOT is active, so use the flags to narrow or disable behavior deliberately.

Use `spring.aot.jpa.repositories.use-entitymanager=true` when managed-type scanning is not sufficient for the JPA repository arrangement and direct `EntityManagerFactory` access is required.

The `spring.aot.data.accessors.*` family controls generated bytecode accessors for domain types, while the `spring.aot.repositories.*` family controls repository source and metadata generation.

These property names are part of the current stable `2025.1.5` / Commons `4.0.5` line. Recheck generated metadata and the current reference docs before carrying them into the upcoming `2026.0.x` train.

Keep the build and runtime target explicit so repository code is validated under the same AOT assumptions.

Enable only the module flags that match the stores the application actually uses.

## Generated output roots

AOT generation can emit repository metadata that includes derived queries, declared query strings, and query method shapes:

```text
# Gradle
build/generated/aotSources/
build/generated/aotResources/

# Maven
target/spring-aot/main/sources
target/spring-aot/main/resources
```

Repository metadata is emitted as generated JSON resources under those output roots, using package-relative locations derived from the source repository type.

Treat generated metadata as build output, not as an API surface to call directly.

Generated AOT repository fragments are internal implementation details. Do not call them directly or build contracts around their generated names.

## Entity or document discovery rule

AOT builds benefit from explicit domain annotations and predictable repository scanning.

```java
@Entity
class Order {
}
```

```java
@Document
class OrderDocument {
}
```

## Decision points

| Situation | Guidance |
| --- | --- |
| JVM deployment without AOT enabled | stay on ordinary repository path |
| JVM deployment with AOT enabled | this reference still applies |
| native-image target is required | review repository patterns for AOT compatibility |
| repository behavior depends on dynamic runtime tricks | expect additional AOT work or avoid that pattern |

## Verification rule

Verify one AOT build emits repository metadata under the generated output root and that the application still starts with the same repository set enabled.
